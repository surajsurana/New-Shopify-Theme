## 1. Audit Summary (Phase 1)

**Current state:** total absence, identical in kind to the Journal's pre-build state — not a weak FAQ, no FAQ at all. Every practical question a bride, bridesmaid, mother or sister might have before her first enquiry — what a corset lehenga actually is, how sizing works without a fitting, how long a made-to-order piece takes, whether international shipping incurs customs duties, how to book an appointment — currently has no self-service answer anywhere on the site. It is either buried inside the product page's per-product accordion (which only ever answers questions about the one garment being viewed, never the brand's policies as a whole) or requires a "cold" WhatsApp message from someone who may not yet feel confident enough to send one.

**Why this matters for a luxury house specifically, not just an SEO one:** at this price point — made-to-order, no ready stock, and (per the now-confirmed FAQ content, §5) ranging ₹30,000 to ₹280,000+ — the practical anxieties that block a first-time visitor from converting are rarely about desire — the photography and product pages already do that work — they're about *risk*: will it fit without trying it on, will it arrive in time, what happens if my measurements change, can I return it if it isn't right. Leaving these unanswered doesn't project exclusivity; it projects opacity, which reads as risk, not prestige. Every benchmark house resolves this without ever looking like a support ticket system.

**Note on price range:** the confirmed FAQ content's price range (₹30,000–₹280,000+) is wider than the figure currently recorded in this project's own CLAUDE.md brand section ("roughly Rs 38,000-80,000+ per piece"). The FAQ content is the more recent, directly-Suraj-confirmed figure (per the Content Strategist's source document, "no invented statistics... every answer is either sourced from confirmed material or explicitly flagged as a gap"), so it's used as-is here — flagged in §7/§9 as a CLAUDE.md housekeeping item, not something this spec resolves unilaterally.

**Benchmark comparison:** Chanel and Hermès both run genuinely spare, typography-led "Client Services" pages — no imagery to speak of, because the register there is quiet reassurance, not seduction, and photography is deliberately reserved for pages whose job is desire. Sabyasachi and other Indian bridal houses that publish shipping/care/ordering FAQ content do so in the house's own editorial voice, never in the cold, boxed, numbered-accordion register of a generic Shopify FAQ app or support-ticket plugin. The failure mode this design deliberately avoids: grey box accordions, a "was this helpful?" thumbs widget, a "submit a ticket" contact form, numbered anchor links — every one of those belongs to e-commerce, not couture, and CLAUDE.md instructs against exactly this register.

**Emotional audit — current state (absence):** Confusion, tending toward mild anxiety for a visitor with a real practical question and no way to resolve it except a message she may not yet feel ready to send. **Emotional audit — redesigned page:** Confidence and Trust. A visitor leaves this page either fully answered, or — for anything the page couldn't resolve — pointed at exactly the right next step (WhatsApp concierge or an appointment), never at a dead end.

## 2. Redesign Concept (Phase 2)

**Naming and framing:** "Frequently Asked Questions" is used plainly as the page's eyebrow label — unlike Journal's "Blog" → "The Journal" reframing, there is no benefit in disguising this page's literal purpose; visitors search for and expect the words "FAQ" / "frequently asked questions," and SEO's own stated goal depends on that literal clarity. The emotional register instead lives in the H1 ("Every Question, Answered Quietly") and the intro copy, not in avoiding the plain label — the same way Chanel's own client-services pages use plain functional labelling with a warm, editorial voice underneath it.

**Why no hero image — a deliberate restraint, not an oversight.** Every other page built so far (Home, About, Client Diaries, Journal) opens on a full-bleed cinematic hero, because each of those pages' job is to build desire or tell a story. This page's job is different: someone arrives here with a specific, practical question and wants to find the answer calmly and quickly — the Chanel/Hermès "Client Services" register, not the "Le Mag"/editorial register. Forcing in a photograph here would do two things wrong: it would slow down exactly the visitor who came here for speed and reassurance, and it would dilute the emotional weight photography carries everywhere else on the site by using it in a context that doesn't call for it. **This page belongs to the same structural family as Collection, Product and Cart** — hero-less, permanently-settled nav, breadcrumb-led — not the same family as Home/About/Journal. This is treated as a considered design decision, and is genuinely useful operationally too: this is the first page in the entire build that requires **zero new photography**.

**Search-within-page, not a filter alone.** With 11 real Q&A pairs across 4 categories — a small, largely fixed content set, unlike the Journal's expected-to-grow article library or the Collection page's full catalogue — a Google-style prominent search box would overstate how much content there is to search through, and a full-viewport search takeover (the pattern already built for global product search, `assets/ka-search.css`) would be the wrong register entirely for searching 11 short text answers. Instead: a single, quiet, underlined italic-serif input sits inline in the page's intro block — visually a smaller sibling of the existing global search overlay's input, not a duplicate of it — doing true text filtering (hides non-matching questions, collapses empty categories, shows a considered no-match state) the moment a visitor starts typing.

**Topic chips are jump-navigation, not a filter — a deliberate divergence from the Journal's own chip pattern.** This is the single most important interaction decision in this spec, and it was arrived at by asking what the Journal's filter-and-hide pattern is actually *for*: managing a content set large enough that showing everything at once would overwhelm the visitor. At 11 items across 4 categories, that condition doesn't hold — hiding three of four categories on a chip click would remove real, harmless browsing value (a visitor reading about sizing might well also want to glance at care instructions two scrolls later) for no corresponding benefit, since nobody is overwhelmed by 11 short questions. So: chips **smooth-scroll to their category** and the currently-active chip **tracks scroll position** as the visitor reads down the page (an in-page contents/tab bar, the way a considered reference document uses running headers) — everything stays reachable, nothing is hidden by default. Search remains true filtering, because searching is explicitly a narrowing action, unlike browsing. Clicking a topic chip also clears any active search, keeping "browse mode" and "find mode" mutually exclusive and easy to reason about rather than a confusing combined filter state.

**Category taxonomy — regrouped in v1.1 to fit the real content, not the reverse.** The v1.0 placeholder draft invented five categories, including a standalone "Appointments & Styling" cluster built around several speculative questions (what to bring to a first appointment, how far in advance to book) that the real, confirmed content simply doesn't contain — the real content mentions booking only once, briefly, inside the sizing/fit answer. Forcing a five-category structure onto 11 real questions would have produced at least one thin, one-question category, which reads as padding, not organisation. The real content instead clusters naturally into four groups, ordered to follow a visitor's actual reasoning sequence: **The Corset, Explained** (what the garment fundamentally is, before anything else) → **Sizing & Occasion** (is this right for my body and my event) → **Ordering & Policy** (how do I get one, what does it cost, what are the rules) → **Shipping & Care** (what happens after I order). This is a better-fitted taxonomy than the placeholder one, arrived at only because real content was available to fit it to — exactly the outcome §5 of v1.0 anticipated when it said the categories should flex to the real content rather than the reverse.

**Inline links within answers — a new requirement the real content introduced.** Two answers (what a corset lehenga is; what makes a corset "structured") link out to the site's two published Journal articles, and one (sizing/fit) links to `/pages/contact` to book a consult. No placeholder-draft answer had contained a link, so this needed a first-time addition: `.ka-acc__inner a` now carries the exact inline-link treatment already used and approved on generic page body copy (`assets/ka-page-generic.css` `.ka-page-generic__body a` — rose-gold, underlined, soft underline offset) rather than inventing a new link style for this one component. See §4.

**Clean question/answer pairing throughout, by design — not incidentally.** Every Q&A unit is one accordion: one button (the complete question, one sentence, one idea), one body (the complete answer, no bundling of a second question into the same block). This was a deliberate constraint during drafting, not a byproduct of using an accordion component — SEO's stated downstream need is FAQPage structured data, which requires exactly this 1:1 mapping, and CLAUDE.md's own instruction was to make sure "the visual structure maps cleanly to a question+answer model, not freeform prose." No schema is written here; the markup is simply kept clean enough that mapping it later is trivial, not an afterthought retrofit.

**Accordion mechanics are a verbatim reuse of the existing product-page component** (`sections/ka-product-main.liquid`, `.ka-accordion`/`.ka-acc`/`.ka-acc__pm`/`.ka-acc__body`/`.ka-acc__inner`) — identical box model, identical max-height open/close animation, identical rotating +/× icon. Only the question-button's *typography* changes (`.faq-acc__head`, a new, minor, documented addition) — the product accordion's head is compact uppercase tracked sans, correct for short compact labels like "Fabric & Embroidery" in a narrow product-page side panel; a full-sentence question reads far better set the way a considered Q&A page actually sets it — quiet italic serif, sentence case, comfortable reading size. This is the same "adapt the mechanism, keep the box model, change only what the new content shape genuinely requires" discipline already used for `.founders` (About Us) and `.j-featured` (Journal) — not a new component invented from nothing.

**First question pre-opened on load.** A page of closed accordion rows reads slightly stark on first arrival; opening the very first question ("What is a corset lehenga?") by default (Chanel and Dior's own client-service pages do the same) shows the format immediately and gives the page some life without adding any new visual noise.

**No dead ends.** A zero-match search state reuses the exact `.ka-collection-empty` visual pattern already approved elsewhere, paired with a direct WhatsApp concierge CTA — "we don't have that answer here" always resolves into a human conversation, never a blank result, consistent with the brand's existing concierge-first ethos (search-404-footer's no-results state, Client Diaries' no-story-match state, product OOS states all follow the same rule).

**Closing CTA copy is reordered relative to every other page's Appointment CTA**, deliberately: this page's typical exit visitor still has one specific unanswered question, so "Ask Our Styling Concierge" (WhatsApp) leads and "Book an Appointment" follows — the reverse emphasis used on Home/About/Journal, where booking usually leads. Matching CTA priority to the visitor's actual state of mind at that specific point in the journey, not applying one fixed CTA order everywhere.

**Pacing:** Breadcrumb (light) → Intro + search (light) → Topic nav (light, sticky) → FAQ list (light) → Appointment CTA (dark image) → Footer (dark) — the same light-throughout-then-one-dark-close rhythm already established for every hero-less page (Collection, Product, Cart), not the hero-then-alternating rhythm used by Home/About/Journal, because this page has no hero to alternate against.

## 3. Section-by-Section Breakdown

**S0 — Navigation.** Global nav, reused verbatim (`sections/ka-nav.liquid`) — see §7 flag F1 for the required nav-state change. **Permanently settled/crystallised state** (solid paper background, dark text, no transparent-over-hero phase) from first paint — same mechanism already used on Collection/Product/Cart via a page-scoped CSS override forcing the nav's "scrolled" visual state (see `assets/ka-collection-breadcrumb.css`'s `#ka-nav` rule), reproduced in the prototype under `#navFaq`. No nav link changes at the top-level desktop nav — see §7 flag F2 for placement decision.

**S1 — Breadcrumb.** *Purpose:* quiet wayfinding under the settled nav, consistent with every other hero-less page. *Structure:* verbatim reuse of `.ka-coll-breadcrumb` (`assets/ka-collection-breadcrumb.css`) — "Home / Frequently Asked Questions." *Responsive:* identical to existing usage, no changes needed.

**S2 — Intro + Search.** *Purpose:* a centred, quiet, typography-led opening that reassures before it asks anything of the visitor, and houses the on-page search field. *Structure:* eyebrow ("Frequently Asked Questions") → two-line italic-serif H1 (small lead-in / large main word, the same proportion pattern already proven on Client Diaries' "Client / Diaries" and Journal's "The / Journal") → one intro paragraph in the house's reassuring voice → search field (`.faq-search`), a scaled-down variant of the existing global search overlay's input (`assets/ka-search.css` `.ka-search-overlay__input`) — same underlined italic-serif visual language, page-anchored rather than a full-viewport takeover, with a small leading magnifier icon and a "Clear" text affordance that appears once text is entered. *Motion:* staggered `.reveal`/`.reveal-delay-1/2/3`, same fadeUp language used sitewide. *Responsive:* type scales down via existing `clamp()` values; search field remains full-width of its centred column at all sizes.

**S3 — Topic Nav.** *Purpose:* in-page contents bar — jump navigation with scroll-tracked active state, not a filter (see §2 rationale). *Structure:* sticky bar, label ("Topics") + 4 chips reusing `.ka-chip` **verbatim** (same visual states — rest/hover/`.is-active` — as every other chip on the site: "The Corset, Explained" / "Sizing & Occasion" / "Ordering & Policy" / "Shipping & Care"), each carrying `data-target` pointing at its category's section id. *Motion/interaction:* click → `scrollIntoView({behavior:'smooth'})` to the target section (which carries `scroll-margin-top` to clear the fixed/sticky header stack) and clears any active search first; scroll position is watched via `IntersectionObserver` to keep the correct chip `.is-active` as the visitor reads down the page. *Responsive:* remains sticky on tablet/desktop; on mobile (≤600px) the sticky `top` offset reduces to sit directly under the (shorter) mobile header, and chips wrap rather than scroll horizontally.

**S4 — FAQ List.** *Purpose:* the page's entire content body. *Structure:* four `.faq-group` sections in fixed order — **The Corset, Explained** (3 questions: what a corset lehenga is, what makes a corset "structured," corset lehenga vs. corset drape saree) → **Sizing & Occasion** (2 questions: sizing/fit and consult booking, occasion suitability) → **Ordering & Policy** (4 questions: price range, made-to-order timelines by category, customization policy, return/exchange policy) → **Shipping & Care** (2 questions: international shipping, care instructions) — each opening with a numbered eyebrow, an italic-serif H2 category title, and one supporting line of copy (the same "even a divider earns a considered line of copy" discipline already used for Journal's category taxonomy) — then a `.ka-accordion` list of that category's questions, using the verbatim-reused accordion mechanics described in §2 with the new `.faq-acc__head` typography. The made-to-order-timelines answer presents its three per-category lead times (Drapes/corset drape sarees 20–25 working days, Indo-Western 30–35 working days, Lehengas 40–45 working days) as a bulleted list using the accordion's existing em-dash bullet style (`.ka-acc__inner ul/li`, already defined for the product-page accordion, reused verbatim) rather than run-on prose, for scannability. Two answers (what a corset lehenga is; what makes a corset "structured") contain an inline link out to the corresponding Journal article; one (sizing/fit) links to `/pages/contact` — see §4 for the link-styling addition this required. *No-results state:* `.ka-faq-empty`, a verbatim visual adaptation of `.ka-collection-empty`, shown only when a search query matches zero questions, pairing a quiet headline/body with a direct WhatsApp concierge CTA. *Responsive:* single column throughout (this page was never a grid), generous but slightly tightened padding/type sizes ≤600px matching sitewide convention.

**S5 — Appointment CTA.** Verbatim reuse of the sitewide `.appointment` component (see homepage-build-spec.md, about-us-build-spec.md, journal-build-spec.md — the fourth page to reuse it unchanged). New copy and a reordered CTA pair only, per §2's "matching CTA priority to visitor state" rationale: "Ask Our Styling Concierge" (WhatsApp) leads, "Book an Appointment" follows.

**S6 — Footer.** Shared global footer, full spec in `Docs/homepage-build-spec.md` §13. New: "FAQ" added to the Maison column, carrying `.is-current` on this page per the existing pattern (see `Docs/about-us-build-spec.md` §6 H3, `Docs/journal-build-spec.md` §3 A7) — see §7 flag F3 for exact placement reasoning.

## 6. Emotional Audit & Luxury Scorecard

**Dominant emotion:** Confidence, moving toward Trust — a visitor arrives with a specific practical uncertainty and leaves either fully answered or clearly pointed at the right human to ask next. Not Awe or Desire, and deliberately so: this page's job was never to build desire (the product/collection/home pages already do that), it is to remove friction and risk from a decision that's already emotionally motivated elsewhere on the site. Confidence is the correct target emotion for a client-services register, the same way it is on Chanel and Hermès's own equivalent pages.

**Luxury Score: 88/100 (v1.1, +1).** Strong: the page never once drops into e-commerce/support-ticket register despite covering genuinely operational content (no grey boxes, no "was this helpful" widget, no ticket form, no numbered anchors); the deliberate no-hero restraint reads as confident editorial judgement rather than a missing asset; the jump-nav-not-filter decision for the topic chips is a considered, content-shape-specific choice rather than a reflexive reuse of the Journal's pattern; the taxonomy now demonstrably fits real content rather than an invented one. Held a few points below the low-90s scores of About Us/Journal specifically because a text-forward support page has a lower emotional ceiling by its very nature than a photography-led editorial page — that gap is expected and correct, not a flaw to fix.

**Editorial Score: 86/100 (v1.1, +2).** Every category divider earns a genuine line of copy rather than a bare label; the real Q&A content itself is confident and unhedged where the brand can be (explicit, unrounded price range, exact per-category working-day timelines) and honest where it should be (customization and returns are stated plainly as case-by-case / generally-not-offered, rather than oversold) — a genuinely more distinctive, harder-to-fake voice than the v1.0 placeholder draft's invented reassurances, now that it's real. ## 7. Shopify Platform Feasibility Flags

### F1 — Template and section needed
**Issue:** No dedicated FAQ template exists; today `/pages/faq` would fall through to `sections/ka-page-generic.liquid`, the deliberately plain soft-launch fallback (confirmed by reading that file directly — it explicitly lists "FAQ" as one of the page types it's a placeholder for).
**Recommendation:** Build a dedicated `templates/page.faq.json` referencing new sections for the intro/search block, the topic nav, and the FAQ list (or one combined section, Technical Director's call), plus reuse of the existing Appointment CTA and Footer sections. Standard Shopify pattern, same approach already used successfully for `templates/page.about-us.json` and `templates/page.client-diaries.json`.
**Risk:** Low.

### F2 — Nav/footer placement — no top-level nav change recommended
**Issue:** Unlike the Journal (which flagged, and left open, a top-level desktop nav addition — see `Docs/journal-build-spec.md` §7 flag J9), this page's content doesn't carry the same "flagship content pillar" weight that would justify permanent top-nav real estate.
**Recommendation:** Footer-only placement (Maison column — see F3), consistent with how Journal's own spec offered a footer-only fallback as the more conservative option. If Suraj later wants FAQ more prominent (e.g. surfaced from the product page's assurance strip, or linked directly from Shipping & Returns / policy pages), that's a low-risk incremental addition, not a structural change to this spec.
**Risk:** None as specified; flagging only so a nav change isn't silently added later without the same explicit review every other shared-chrome change in this project has received (per CLAUDE.md's own standing convention).

### F3 — Footer column placement
**Issue:** The footer has no dedicated "Support" or "Help" column — only Collections, Maison, Contact, and the newsletter column exist today (confirmed directly in `sections/ka-footer.liquid`).
**Recommendation:** Add "FAQ" to the existing Maison column's fallback link list (alongside About, Client Diaries, Journal, Visit Us, Book an Appointment, Concierge), matching exactly how Journal was added there (`Docs/journal-build-spec.md` §3 A7). Also add the corresponding entry to the real `footer-the-house` menu in Shopify Admin if one is already configured there, since the code fallback only renders when that menu is empty (same caveat already documented for the Journal link).
**Risk:** Low — an addition to an existing column, not a new column or layout change.

### F4 — Sticky topic-nav offset must be confirmed against live CSS
**Issue:** §4 flags that the prototype's `top: 104px` value for `.faq-topics` is reasoned from precedent (matching Cart/Product's sticky-element family) rather than measured directly from a rendered page, since this task's environment only has access to the live published site, not the in-progress draft theme.
**Recommendation:** Technical Director should verify the real combined height of the utility bar + settled nav on a live hero-less page (Collection/Product/Cart) and adjust `.faq-topics`'s `top` value to match exactly, rather than trusting 104px blindly.
**Risk:** Low — a pixel-offset detail, not a structural risk. Worst case if wrong: a few pixels of visual overlap/gap under the sticky bar, easily caught in QA.

### F5 — Search is pure client-side text filtering, no backend/API dependency
**Issue:** Unlike the site's global product search (which depends on Shopify's Predictive Search API — see `Docs/search-404-footer-build-spec.md` §1.8 flag S1), this page's search operates over a small, fixed, page-local content set (11 Q&A pairs).
**Recommendation:** Implement as plain client-side JavaScript filtering over the rendered DOM (exactly as prototyped) — no Shopify API, no app, no server round-trip needed. This is simpler than the global search build, not a scaled-down version of it.
**Risk:** None. Standard, well-understood client-side pattern; the prototype's implementation is directly portable to Liquid-rendered markup.

### F6 — FAQPage structured data is SEO's build, not this spec's
**Issue:** The routing brief notes a real FAQ page needs to exist before FAQPage schema can be added, and asked this spec to ensure the visual structure maps cleanly to a question+answer model.
**Recommendation:** This spec deliberately does not write or specify JSON-LD/schema markup — that remains SEO & Discoverability's task now that real content exists. What this spec does guarantee: every Q&A unit in the final markup is one unambiguous question (accordion button text) paired with one unambiguous complete answer (accordion body content), with no bundled multi-question blocks — confirm with SEO's team that this mapping is sufficient for their schema needs before Technical Director builds, since a schema-authoring pass may want specific `id`/`data-` attributes added to the markup that this design-only spec hasn't anticipated.
**Risk:** Low, but flagging explicitly as a coordination point between this task and SEO & Discoverability's follow-on work, not something to silently assume is already covered.

**Risk:** Low technically, but a real content-accuracy risk if skipped — flagging because it's exactly the kind of small, easy-to-miss cross-page dependency this project's QA gate (CLAUDE.md, "Standing rule: QA gate is mandatory") exists to catch.

**Overall: Simple.**

- **Zero new photography, zero new colour/type/spacing tokens, zero new third-party dependencies.** The highest-reuse page built in this project to date (§6).
